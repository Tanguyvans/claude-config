## 2026-04-22

- [bug] Le SKILL.md dit "python3 -m http.server dans le dossier **OU** ouvrir directement Carrousels.html dans un navigateur moderne" — c'est faux. Ouvrir via `file://` casse systématiquement parce que Babel standalone ne peut pas fetch les `.jsx` externes par CORS (page blanche). Remplacer par : `python3 -m http.server` est **obligatoire**, `file://` ne fonctionne pas.
- [manque] Aucune instruction pour l'export PDF. Ajouter au template un mode `?print=1&format=instagram|tiktok|linkedin` qui bypass le DesignCanvas, stacke les slides à taille native avec `page-break-after: always`, et injecte `@page { size: WxHpx; margin: 0 }`. Ensuite `Cmd+P → Save as PDF`. A dû être ajouté ad-hoc.
- [manque] Pour les carrousels "news release" sur un produit/marque (Google, OpenAI, Anthropic, etc.), le workflow produit quelque chose de trop générique parce qu'il s'appuie sur `<SketchImage>` placeholder et pas de logos de marque. Ajouter une directive : pour les news d'un produit spécifique, intégrer systématiquement (a) les logos de marque en inline SVG sur la cover + slide features, (b) un mockup stylisé de l'UI du produit sur la slide démo plutôt qu'un `<SketchImage>` vide. L'utilisateur a explicitement dit "ça fait trop générique" après la v1.
- [amelioration] La règle "ne jamais lancer de serveur" est stricte par défaut, ce qui est OK, mais dans la réponse de livraison il vaut mieux proposer activement à l'utilisateur "dis-moi si tu veux que je lance `python3 -m http.server` pour toi" plutôt que de le laisser se débrouiller. Economise un aller-retour "je ne vois rien".
- [positif] Le système de primitives (`SlideFrame`, `SketchBox`, `Highlight`, `SlideHeader`, `FORMATS`, `DISPLAY_SCALES`) plus les 3 exemples archétypes (news/tuto/versus) permet de générer un carrousel complet en un seul passage Write, sans réimplémenter de CSS/composants. Garder ce pattern.

## 2026-04-23

- [amelioration] Pour les carrousels "versus" entre deux modèles (ex: GPT 5.5 vs Claude Opus 4.7), le SKILL.md ne met pas en garde contre la confusion **model-level vs tooling-level**. J'ai glissé plusieurs fois dans des comparaisons d'écosystème (Routines, Memory tool, Claude Code) au lieu de rester sur les capacités du modèle lui-même. Ajouter dans la section "versus" : "Pour un versus entre 2 modèles : claims strictement model-level (modalités, contexte, benchmarks, tokenizer, effort levels, instruction-following). Les features tooling (SDK, features plateforme, cron jobs, etc.) appartiennent à l'écosystème — à éviter ou à étiqueter explicitement 'côté plateforme'."
- [amelioration] Pour les carrousels "news/versus" sur des modèles avec benchmarks, aucune étape de **fact-checking explicite** avant d'écrire les slides. J'ai produit plusieurs claims erronés (context GPT "~400k" alors que c'est 1M, etc.) qui ont dû être corrigés après coup par l'utilisateur. Ajouter une étape 2.5 au workflow : "Avant Write, vérifier chaque chiffre/fait via WebSearch ou WebFetch sur la doc officielle. Pour un versus, croiser au moins 2 sources indépendantes pour les benchmarks."
- [manque] **Export PNG via Playwright** : j'ai dû créer `export.html` + `export.mjs` ad-hoc pour sortir 10 PNG (5 slides × 2 formats) en 2160×2700/2160×3840. Ajouter à `assets/` un `export.html.template` et un `export.mjs` réutilisables — ils reprennent les composants sans DesignCanvas, forcent `DISPLAY_SCALES[format] = 1`, et screenshootent chaque slide à résolution native × 2 (retina). C'est très demandé et ça résout le problème d'export pour IG/TikTok proprement.
- [amelioration] Le slide CTA de base (3 tags simples `<SketchBox>`) est visuellement pauvre. L'utilisateur a explicitement dit "c'est pas très beau". J'ai dû le refaire avec 3 cartes colorées + emoji grands + rotations + signature "on se revoit →". Mettre à jour l'exemple CTA dans `examples/*.jsx` avec ce pattern plus punchy (cartes couleur / icône géant / rotation / signature hand-drawn finale).
- [positif] Le pattern de réduction de 7 → 5 slides via fusion de deux slides en un composant partagé (`C1_ForcesLimites` avec 2 colonnes symétriques) a très bien fonctionné. Pour un versus équilibré, le pattern "1 pro + 1 con par modèle côte-à-côte" est plus punchy que 2 slides séparées. À mentionner comme variante dans la structure narrative.
